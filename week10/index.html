Client-side form validation

---------------------------------------------------------------------------------------------------

Client-side form validation is where you ensure all form controls are filled out properly before submitting
the data to the server. This helps to reduce delays by letting the user fix any issues before the server
finds it and sends it back to be fixed by the user.

Client-side form validation should not be the only form of validation and security. You should always do server
side validation as well because client side validation is easy to bypass.

---------------------------------------------------------------------------------------------------

Form Validation happens in ways like making a field required or making it numerical only and accepting 10 values
for things like phone numbers. You can also require specific formats for things like email among others. If the
forms are filled out correctly then it allows the application to submit the data to the server.

Built in form validation uses HTML5, performs better and requires less work but is not very customizable.
Things like required, min, max are built in form validation. It is a good idea to style validation errors
so that they are visible to the user so they can fix them. Example:

<label for="choose">Would you prefer a banana or cherry? (required)
 <input id="choose" name="i_like" required>

---------------------------------------------------------------------------------------------------

JavaScript is more customizable but you have to create it all.

Constraint validation API which most browsers support uses methods and properties from the form element DOM interfaces
HTMLButtonElement, inputElement and selectelement are just a few.
the validationMessage will return a message telling you that something isn't correct. so you would correlate the correct
validitystate with your element. toolong, maxlength, tooshort etc...

The nice thing about constraint validation api is it allows for more customization or flexibility. You can stylize the auto
generated error messages using the api where normally they are not customizable.

---------------------------------------------------------------------------------------------------

At the end of the day there is no true answer for which is the best kind of validation. It is up to you, what you prefer
and what best fits your situation.

Using the Fetch API

---------------------------------------------------------------------------------------------------

Fetch API gives JavaSCript a handle to grab on to to be able to manipulate HTTP like requests and responses.
This uses the fetch() method fetch() won't send cross-origin cookies unless you set the credentials init option.
Since April 2018. The spec changed the default credentials policy to "same-origin"
fetch('some url example')
.then()
.then()

Fetch can accespt an init object as a second paramter like grab a specific header, bod, mode or credentials. You could
even exclude something using omit.

https://developer.mozilla.org/en-US/docs/Web/API/fetch
---------------------------------------------------------------------------------------------------

You should check for that the fetch was successful. You can do this with an if statement to see if the response
is not ok, then show a message that response was not ok.
---------------------------------------------------------------------------------------------------

Headers can be guarded. basically means you cant change them inside your fetch request
